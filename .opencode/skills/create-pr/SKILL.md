## What I do

Open a pull request from a feature branch into `develop` with a title and body that match the repo's conventions and that will pass CI on the first run.

## When to use me

Activate whenever the user asks to open, create, or submit a pull request. **Never** target `main` from this skill — `main` is release-only and the release PR is opened by semantic-release, not by hand. To cut a release, use the `create-release` skill.

## Pre-flight

Before opening the PR:

1. `git status` — clean working tree, or note what's untracked.
2. `git diff <base>...HEAD` — the full PR diff. Read it end to end. Don't open a PR on a diff you haven't inspected.
3. `git log --oneline <base>..HEAD` — every commit that will land.
4. `git rev-parse --abbrev-ref HEAD` — confirm you're on a feature branch, not on `develop` or `main`.
5. `gh pr list --head <branch> --state open` — confirm there's no existing PR for this branch.

If the diff contains secrets, generated files, or `dist/`, stop and ask the user. Stage only intended changes.

## Local validation

Run the same checks CI runs, in this order. **Do not open a PR if any step fails** — fix the issue first.

```bash
pnpm install --frozen-lockfile
pnpm run lint
pnpm run format:check
pnpm test
```

If `format:check` fails, run `pnpm format` and re-stage the rewritten files. Do not commit the format changes as a separate `style:` commit — fold them into the existing commits or amend the relevant commit (only if local-only; never amend a pushed commit).

## Branch and target

- **Target branch: `develop`**. This is the default branch per `ROADMAP.md` and the only branch the CI workflow listens for (`on: pull_request: branches: [develop]`).
- **Source branch**: the user's current feature branch, e.g. `feat/aggregator-byTool` or `fix/trace-error-sessionid`.
- If the user is on `develop` or `main`, ask them to create a feature branch first.

## Title and body

### Title

Conventional commit format, same rules as the `create-commit` skill:

- Lower-case subject.
- Max 100 characters.
- Format: `<type>(<optional-scope>): <subject>`.
- Imperative mood.

`gh pr create --title "feat(aggregator): add byTool dimension to MetricsSnapshot" ...`

### Body

Include:

1. **What** — one-paragraph summary of the change and why.
2. **How** — bullet list of the key implementation choices.
3. **Tests** — note which tests were added and what they cover.
4. **Checklist** — copy-paste this:
   - [ ] `pnpm run lint` passes
   - [ ] `pnpm run format:check` passes
   - [ ] `pnpm test` passes
   - [ ] `ROADMAP.md` updated if this closes a phase
   - [ ] `CHANGELOG.md` will be regenerated by release-please (do not edit manually)

## CI status

After opening the PR, monitor the `ci` workflow:

```bash
gh pr checks --watch
```

If CI fails, push follow-up commits to the feature branch — never force-push, never push directly to `develop`.

## Forbidden

- Targeting `main`. Use `create-release` for releases.
- Force-pushes to the feature branch (`--force` or `--force-with-lease` only with explicit user approval).
- Direct pushes to `develop` or `main` from the agent.
- Disabling or skipping CI checks.
- Drafts that are never flipped to ready — if the user asks to "open a PR", open it ready for review unless they say "draft".
